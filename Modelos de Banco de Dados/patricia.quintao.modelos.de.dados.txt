	Modelo de dados:
			- Um modelo de dados é um conjunto de conceitos que podem ser utilizados para descrever a estrutura lógica e física de um BD.
Os modelos de dados podem ser divididos em CONCEITUAIS,REPRESENTATIVOS e FÍSICOS.
	Os modelos de dados conceituais apresentam os dados da forma como os usuários finais percebem. Fornecem uma visão mais próxima do modo como os
usuários visualizam os dados realmente. O exemplo é o MODELO DE ENTIDADE RELACIONAMENTO (MER).
	Os modelos de dados representativos ou lógicos ou de implementação : 
		- São introduzidos informações sobre a forma pela qual os dados são armazenados dentro de um computador.
			Exemplo: Grande exemplo temos o MODELO RELACIONAL (Tabelas)
	Modelos físicos: Conhecido como de BAIXO NÍVEL. Oferecem uma visão detalhada do modo como os dados estão realmente armazenados no PC.
	** LEVA EM CONSIDERAÇÃO TODAS AS REGRAS E LIMITAÇÕES DE BD.** Contém a descrição da implementação da base de dados assim como são descritas
as estruturas de armazenamento e os métodos de acesso.
	Outros modelos pouco utilizados:
	- Modelo Hierárquico ou de árvore - 
	- Árvore composta por nós. O nível mais alto da árvore denomina-se RAIZ. TOPO
	- Cada nó representa um REGISTRO com seus CAMPOS. 
Sua estrutura então é uma coleção de registros que são conectados uns aos outros por meio de ligações.
Sendo que cada REGISTRO é uma coleção de CAMPOS(atributos), os quais contêm apenas um valor de dado.
		*** UM REGISTRO PAI PODE TER VÁRIOS FILHOS, MAS UM FILHO SÓ PODE TER UM PAI. ***
	Obs: Os dados são organizados de cima para baixo ou em estrutura de árvore invertida.
** É O QUE MAIS SE ADAPTA A SITUAÇÕES NAS QUAIS AS RELAÇÕES LÓGICAS ENTRE OS DADOS PODEM SER REPRESENTADAS COM A ABORDAGEM(UM-PARA-MUITOS). ***
	Modelo em Rede:
			- Um modelo em rede é uma extensão do modelo hierárquico. Elimina-se o conceito usado de HIERARQUIA e permite que um mesmo
registro esteja envolvido em várias associações.
	- Ao invés de usar relações de cardinalidades um-para-muitos, o modelo em rede é uma relação MEMBRO-PROPRIETÁRIO, na qual um membro pode ter
muitos proprietários.
	MODELO RELACIONAL:
		-O modelo relacional se tornou o mais popular. Usa-se um formato TABULAR PADRÃO. Os elementos são localizados em tabelas BIDIMENSIONAIS.
E as tabelas são organizadas em linhas e colunas. 
É utilizado em operações de ÁLGEBRA RELACIONAL para a manipulação de dados: 
	- Alguns exemplos são SELEÇÃO,PROJEÇÃO E AGRUPAMENTO.

- Os modelos considerados REPRESENTACIONAIS - implementação/lógico - são os modelos em REDE,HIERÁRQUICO E RELACIONAL.

A modelagem conceitual é uma fase muito importante no projeto de uma aplicação de BD.
Um modelo conceitual de dados é aquele em que os objetos, suas características e relacionamentos possuem uma representação fiel ao ambiente.
O modelo conceitual independe de HARDWARE e SOFTWARE. Ou seja, não depende de nenhum SGBD para implementá-lo.
O esquema conceitual é uma descrição concisa dos requisitos de dados dos usuários e inclui detalhes dos tipos de ENTIDADES,RELACIONAMENTOS e RESTRIÇÕES.
Sendo assim, permite que PROJETISTAS de BD se concentrem em especificar as PROPRIEDADES DOS DADOS. Sem se preocupar com detalhes de armazenamento e implementação.

Alguns conceitos de modelagem de BD:
	- BOTTOM-UP: Inicia a partir dos conceitos mais detalhados percorrendo o sistema até os mais abstratos.(alto nível de visão)
	- Top-DOWN: Dos conceitos mais altos de abstração aos mais detalhados.
	- INSIDE-OUT: Inicia-se dos conceitos mais relevantes adicionando conceitos secundários. O modelo E-R usa-se dessa técnica.
		( Identifica-se o núcleo do sistema e gradativamente acrescentam-se novas entidades. )

MODELO ENTIDADE-RELACIONAMENTO (PRINCIPAL EXEMPLO DE MODELO CONCEITUAL) Peter Chen apresenta uma notação gráfica para diagramas com poucos conceitos,
modelo simples e representação gráfica de fácil compreensão.
O MER ou MODELO ER é um modelo de dados conceitual de alto nível, cujos conceitos foram projetados para estar o mais próximo possível da visão que o
usuário tem dos dados. Não se preocupando em representar como estes dados estarão realmente armazenados.
	É usado em projetos de BD com representação gráfica pelo DER (diagrama de Entidade-Relacionamento)
O modelo é abstrato e define quem são Entidades,relacionamentos e atributos e cardinalidades. Para representar de forma visual temos o DER.
	As bancas gostam de colocar que o MER é um diagrama sendo que a sigla correta é DER para o diagrama.

ENTIDADES:
	- Objeto básico tratado pelo modelo Entidade-Relacionamento. É um objeto do mundo real,concreto(pessoas,carro,funcionário)ou abstrato
(um projeto,curso,disciplina) e que possui existência ---->independente.<---
		Quando usamos a entidade e referimos ela no diagrama damos o nome de INSTÂNCIA(OCORRÊNCIA).

ATRIBUTOS:
	- São propriedades particulares que descrevem uma entidade e características de um relacionamento também.
Exemplo: Entidade CARRO possui diversos atributos como: MARCA,MODELO,COR,FABRICANTE,CHASSI.
Exemplo: Entidade EMPREGADO pode ter atributos: SETOR,RG,SALÁRIO.
Exemplo: Entidade FILME pode ter atributos: TÍTULO,ANO,TAMANHO,GÊNERO,CLASSIFICAÇÃO
* Detalhe: No modelo físico os atributos são chamados de COLUNAS ou CAMPOS. *
** Detalhe 02: Todo ATRIBUTO deve possuir um DOMÍNIO. Ou seja, um conjunto de possíveis valores para o atributo. A qual no final deve determinar um valor
a receber, necessariamente.
*** Se não possuir um domínio, certamente ele será nulo ou desconhecido valor. Caso algum atributo não se aplicar. ***
Atributos Simples (atômico):
	-São atributos não divisíveis. Exemplo: CEP
Atributos Compostos:
	-Podem ser divididos em atributos básicos mais simples com significados independentes.
	Exemplo: Atributo ENDEREÇO pode ser dividido em número,logradouro,cidade,estado ou CEP.
Atributos MONOVALORADO:
	- Possuem apenas um valor para uma entidade em particular.
Atributos Multivalorados:
	- Possuem múltiplos valores. (Uma única entidade possui diversos valores para este atributo.) Representado por um desenho de uma ELIPSE com LINHA DUPLA.
Exemplo: O atributo TELEFONE pode possuir vários números de telefones para a entidade PESSOA.
Exemplo02: O atributo IDIOMA de uma entidade ALUNO pode conter INGLÊS E FRANCÊS ou vários idiomas além do INGLÊS E DO FRANCÊS,ALEMÃO,JAPONÊS.
ATRIBUTOS ARMAZENADOS:
	- Aqueles que realmente pretendemos guardar no BD.
ATRIBUTO DERIVADO:
	- Podem ser gerados a partir de outros atributos. Por exemplo: O atributo DATA_NASCIMENTO pode derivar outro atributo chamado IDADE.
ATRIBUTO CHAVE:
	- é aquele atributo determinante. é um atributo único para cada elemento do conjunto, servindo para identificar UNIVOCAMENTE UM ELEMENTO.
Exemplo: O atributo CPF pode ser um atributo determinante para a entidade PESSOA, pois diferencia cada funcionário de forma única dentro do conjunto.
Já que 'DOIS FUNCIONÁRIOS NÃO PODEM TER O MESMO CPF'.

Além disso,os atributos são classificados em :
	ATRIBUTO DESCRITIVO: -  Descrevem e representam características de um objeto.
	ATRIBUTO NOMINATIVO: -  Além de descrever, também definem nomes ou rótulos de identificação dos objetos aos quais pertencem.
				Exemplo: Código,matrícula,número.
	ATRIBUTO REFERENCIAL: - Faz referência a outra entidade. Exemplo: Chave estrangeira.

RELACIONAMENTOS:
	-	Um relacionamento pode ser entendido como uma associação entre instâncias de entidades devido a regras de negócio.
São relações entre duas ou mais entidades, determinando uma associação entre as mesmas.
(representado por um LOSANGO)- notação.

GRAU DE RELACIONAMENTO:
	- A primeira característica de um relacionamento é que este tem um grau. É o número de entidades que fazem parte desse relacionamento.
* UMA ENTIDADE PODE PARTICIPAR DE UM RELACIONAMENTO COM QUAISQUER OUTRAS ENTIDADES DO MODELO, INCLUSIVE COM ELA MESMA. (AUTO-RELACIONAMENTO)(UNÁRIO)
Exemplo: Um supervisor deve-se também se supervisionar.
Exemplo: Uma instância ocorre quando o papel de esposa e marido ocorrem para o relacionamento CASAMENTO.
É também chamado de RELACIONAMENTO RECURSIVO - AUTORRELACIONAMENTO - RELACIONAMENTO UNÁRIO.
 - São casos especiais em que uma entidade se relaciona com si própria (DEVE ter um RELACIONAMENTO INCLUSO)
Exemplo: Uma entidade FUNCIONÁRIO pode assumir o papel de gerente, coordenador ou supervisor.

Relacionamentos Binários: 
			- Duas entidades
Relacionamentos Ternários:
			- Três entidades
Relacionamentos N-ários : 
			- Acima de duas entidades.

Observações sobre os atributos descritivos: 
	Vale lembrar de que maneira geral os atributros descritivos descrevem as características e representam um objeto.
	Pode ser assim com RELACIONAMENTOS também, além das ENTIDADES.
		Um relacionamento pode possuir um atributo em especial. Isso ocorre quando há um relacionamento N:N que necessita ser 'quebrado' em dois
								relacionamentos de 1:N.
	Exemplo bem claro:
			- Um médico atende vários pacientes e um paciente é atendido por vários médicos. (n:n)
			- Podemos então usar um relacionamento entre as entidades MÉDICO E PACIENTE.
			- Sendo assim a consulta pode ter o RELACIONAMENTO BINÁRIO e ainda possuir atributos como a data_consulta,valor_consulta e exames.

Cardinalidade de Relacionamento:
	- Procura definir o número de ocorrências ou instâncias de uma entidade se relaciona com a outra, tendo um mínimo e um máximo.
Cardinalidade Mínima: 
	- Define se o relacionamento entre duas entidades é obrigatório ou não. 
	- Usada também para definir se é parcial ou opcional ou total/obrigatória.
PARCIAL/OPCIONAL:
	- Uma instância para uma certa ENTIDADE PODE OU NÃO participar de um relacionamento.
	- Ela é OPCIONAL se a cardinalidade mínima for 0 zero, uma associação opcional.
	- Ela é parcial se for igual ou maior que 1. 
Por exemplo: Departamento (1.1)----------Alocação---------(10.N)Empregado
-Devem existir pelo menos 10 empregados alocados = Parcialidade mínima.

Cardinalidade de Atributos:
	- Um atributo pode possuir uma cardinalidade que faz parte da entidade. Irá definir quantos valores esse atributo pode estar associado a u
instância de uma entidade ou relacionamento a qual pertence.

Exemplos: (1.1) - Obrigatória - Como é mínimo e máximo uma única vez. Não precisa ser representada no diagrama.
(0.1) - Opcional - 
(0,N) - Opcional e multivalorada(n)

Conceitos de GENERALIZAÇÃO/ESPECIALIZAÇÃO:
	- Generalização: Em que se cria um conjunto de entidades genérico contendo características em comum.
	- Especialização: Apresentam diferenças nas características das entidades.

Por exemplo:  			Entidade MÉDICO é uma GENERALIZAÇÃO.
				Se modelarmos os dados, podemos usar a ESPECIALIZAÇÃO. Ocorre quando a ENTIDADE MÉDICO pode se ESPECIALIZAR em DUAS ENTIDADES:
		- Médico Residente [ENTIDADE]
		- Médico Efetivo   [ENTIDADE]
				A especialização podem ser divididas em categorias,cada qual com seus atributos específicos.
Assim, na especialização, podem possuir atributos em comum e atributos distintos. 
	obs: O TRIÂNGULO representa uma associação de ESPECIALIZAÇÃO/GENERALIZAÇÃO.
	OBS02: A entidade especializada HERDA as propriedades da entidade genérica.
Exemplo:
	- A entidade CLIENTE possui seus atributo chave código e nome, logo a associação (representada por um triângulo) leva a especialização/generalização
Significa que a entidade Cliente foi especializada em duas outras entidades - [pessoa física][pessoa jurídica]
(sendo que a entidade especializada HERDA as propriedades da entidade genérica)
 	A ESPECIALIZAÇÃO É O PROCESSO DE DEFINIR UM CONJUNTO DE SUBCLASSES DE UM TIPO DE ENTIDADE. A ENTIDADE QUE FOI ESPECIALIZADA É CHAMADA DE SUPERCLASSE.
A GENERALIZAÇÃO É O PROCESSO DE ENCONTRAR CARACTERÍSTICAS COMUNS DE ALGUMAS ENTIDADES E CRIAMOS UMA SUPERCLASSE PARA ELAS.

Restrições PARCIAL:
	- Nem toda ocorrência da entidade genérica possui uma ocorrência(instância) correspondente em uma entidade especializada.
Exemplo:Entidade FUNCINÁRIO está associado pela especialização ENTIDADES: - Motorista e - Secretária. 
Na GENERALIZAÇÃO/ESPECIALIZAÇÃO PARCIAL: Nem todo funcionário é motorista e nem todo funcionário é secretária.
obs: PODE HAVER FUNCIONÁRIOS QUE NÃO SEJAM NEM MOTORISTAS E NEM SECRETÁRIOS.

RESTRIÇÃO TOTAL:
		- Para toda a ocorrência da entidade genérica existe sempre uma ocorrência em uma das entidades especializadas.
Exemplo:
	- Entidade CLIENTE indica que é PESSOA FÍSICA OU PESSOA JURÍDICA. Obrigatoriamente deverá ser uma entidade ou a outra entidade.	
	

Exemplo:
	Uma ENTIDADE DEPARTAMENTO pode ter no mínimo nenhum (0) empregado em sua alocação. Assim pode-se definir que pode ter um 
minimo opcional. Indicando assim, que podem existir departamentos que não tem nenhum empregado relacionado a ele.
TOTAL/OBRIGATÓRIO:
	A cardinalidade mínima recebe uma associação obrigatória que indica que o relacionamento DEVE obrigatoriamente associar a uma instância de entidade.
Com no mínimo 1, (um). Ou seja, uma entidade DEPARTAMENTO deve possuir pelo menos 1 (um)RELACIONAMENTO para a entidade EMPREGADO alocado,(1).

CARDINALIDADE MÁXIMA: Número máximo de instâncias da entidade associada que devem se relacionar a outra entidade.
	Em um determinado contexto temos: 
			Entidade: Estúdio ----- gerencia(relacionmento)--------Entidade: Filme
Podemos definir o número de vezes que a entidade FILME pode associar a um ESTÚDIO(ENTIDADE). A qual é somente 1 filme.
Já a entidade ESTÚDIO PODE gerenciar vários filmes, ou seja, sua associação é de cardinalidade máxima (n) para várias entidades FILMES.
Sendo assim:
			[Entidade: Estúdio (1)-------- gerencia-------(n) Entidade:Filme]
					|			       ^
					|			       |		
					-------------------------------
Conclui-se que o Estúdio gerencia diversos filmes (n)[Cardinalidade Máxima] e Filme pode pertencer a somente (1) estúdio.
	*** ATENÇÃO!!! a CARDINALIDADE de uma entidade vai anotada do outro lado após o relacionamento, assim como a outra ENTIDADE, sua cardinalidade 
							é do outro lado após o relacionamento. ***

Questão concurso: CESPE-2008
	Em um diagrama entidade-relacionamento, a cardinalidade representa o número de ocorrências das entidades em um dado relacionamento. [correto]
	Explicação: A questão esta correta. Para definir o número de ocorrências de uma entidade usamos o conceito de cardinalidade, que indica
quantas ocorrências de uma entidade participam no mínimo e no máximo de um relacionamento.
Exemplo: Uma entidade EDITORA_de_LIVROS -(1,1)-------------------Publica(relacionamento)--------------------(0,n) Livros
		Uma editora publica 'n' livros. Já a entidade LIVRO é obrigatoriamente publicado por no máximo 1(uma) editora.
		A cardinalidade mínima ZERO representa que o relacionamento é OPCIONAL ou PARCIAL. ( ou seja, a editora pode publicar o livro ou não )
		Obs: Se a cardinalidade mínima for representada com um número maior ou igual a zero, o relacionamento é OBRIGATÓRIO. 
				Sendo assim, a ENTIDADE EDITORA DEVE publicar pelo menos 1 livro.
Obs: N é um número arbitrário que representa qualquer valor maior que 1.

Mais exemplos:
	(0:N) - Um empregado pode trabalhar em nenhum ou muitos projetos.
	(N:N) - Um empregado pode trabalhar em muitos projetos e um projeto tem vários empregados.
	(1:1) - Uma Entidade A está associada no máximo a uma, somente, entidade em B e a Entidade B está associada no máximo a uma entidade em A.
			[A-----------------B]
			[B-----------------A]
			
	(1:M) - Uma entidade em A está associada a qualquer núnmero de entidade em B. Enquanto uma entidade em B está associada no máximo a uma(1)
 entidade em A.
	(M:M) - Uma entidade em A está associada a qualquer núnmero de entidade em B e uma Entidade em B está associada a qualquer número de entidades em A.
			OBS: O NÚMERO DE ENTIDADES QUE PARTICIPAM DE UM TIPO DE RELACIONAMENTO É IRRESTRITO 
				E ARMAZENAM MUITO MAIS INFORMAÇÕES DO QUE DIVERSOS RELACIONAMENTO BINÁRIOS.

ENTIDADE FRACA:
		- É uma entidade que tem uma relação de dependência com outra entidade.
		[A ENTIDADE FRACA SÓ EXISTE SE EXISTIR A ENTIDADE COM A QUAL ESTA RELACIONADA]
São representadas por losangos duplos.
Exemplo: A entidade funcionário possui a entidade DEPENDENTES com seus atributos.
FUNCIONÁRIO-------------POSSUI-------------------------DEPENDENTE <--- Entidade Fraca

Nas entidades fracas, sempre teremos uma participação total da entidade fraca. Um elemento da entidade fraca tem que estar associado a, pelo menos,
um elemento da entidade normal.
Existe também o relacionamento entre entidade fraca e a entidade normal chamado de RELACIONAMENTO IDENTIFICADOR. [Triângulo duplo]

ENTIDADE ASSOCIATIVA:
		- Relacionamento: É uma associação entre entidades.
Na modelagem ER não é prevista a possibilidade de associar a uma entidade com um relacionamento.Ou de associar dois relacionamentos entre si.
Em alguns casos é desejável permitir uma associação entre uma entidade e um relacionamento, gerando uma ENTIDADE ASSOCIATIVA.
Por exemplo:
		- Temos a ENTIDADE MÉDICO--(N)-------- CONSULTA-------(n)Paciente
Precisamos modelar o diagrama e colocar a entidade MEDICAMENTOS, RECEITADOS AOS PACIENTES.
A solução é então transformar o relacionamento entre Médico e Paciente numa ENTIDADE ASSOCIATIVA e relacioná-la a ENTIDADE MEDICAMENTOS através de um outro
relaciomento PRESCRIÇÃO.

DER - Diagrama Entidade Relacionamento:

RELACIONAMENTOS.
	- O DER fornece uma visão lógica do BD. Proporciona um conceito mais generalizado de como estão estruturados os dados de um sistema.

e) será necessária uma entidade associativa entre Cliente e Conta.

Obs: Para definir o número de ocorrências de uma entidade usamos o conceito de CARDINALIDADE.<-- Indica quantas ocorrências de uma entidade participam 
no mínimo e no máximo do relacionamento. Em outras palavras, a cardinalidade de um relacionamento expressa quantas entidades de um grupo se relacionam
como uma entidade do outro por instâncias.(ocorrências)
Cardinalidade Mínima:
	- Define-se o relacionamento entre duas entidades é OBRIGATÓRIO ou NÃO.
É O NÚMERO MÍNIMO DE INSTÂNCIAS DA ENTIDADE ASSOCIADA QUE DEVEM SE RELACIONAR COM UMA INSTÂNCIA DA ENTIDADE EM QUESTÃO.
A participação na cardinalidade mínima pode ser: 
			PARCIAL/OPCIONAL OU TOTAL/OBRIGATÓRIA
PARCIAL ou OPCIONAL:
	- Uma (instância) ocorrência da entidade pode ou não participar de um determinado relacionamento.
	- É indicado pela cardinalidade = 0 (zero)
	- A cardinalidade mínima zero recebe a denominação de 'associação opcional'. Uma vez que indica que o relacionamento PODE OU NÃO associar uma
ocorrência de entidade a cada instância da outra entidade em questão.

Exemplo:
	DEPARTAMENTO-(1.1)-----------ALOCAÇÃO--------------(0.N)-EMPREGADO

A ENTIDADE DEPARTAMENTO pode ter no mínimo nenhum empregado (0), e no máximo, vários empregados.(n)
Obs: Indica que podem existir departamentos que não tem nenhum empregado relacionado a ele.

TOTAL ou OBRIGATÓRIA:
	- Quando todas as instâncias (ocorrências) de uma entidade DEVEM participar de um determinado relacionamento.
	- Indicado com a cardinalidade mínima maior que zero... 1
	- A cardinalidade mínima (1) recebe a denominação de 'associação obrigatória', uma vez que indica que o relacionamento DEVE obrigatoriamente
associar uma ocorrência (a uma instância) de entidade a cada instância de outra entidade.
Exemplo:
	ENTIDADE DEPARTAMENTO-(0.0)--------------->ALOCA-------------------(1.N)->EMPREGADO
** Obs: Todos os departamentos devem possuir pelo menos(no mínimo) um empregado alocado.(1)
	NÃO PODERÁ EXISTIR QUE UM DEPARTAMENTO NÃO TENHA NENHUM EMPREGADO.

Cardinalidade Máxima de uma entidade: número máximo de instâncias da entidade associada que devem se relacionar com uma instância da entidade.
Em modelos de ER geralmente mostram a cardinalidade máxima. Na ordem. ESQUERDA PRA DIREITA. ATENÇÃO.

Na prática, há dois tipos de cardinalidade:
	- CARDINALIDADE MÍNIMA: - Opcional: indica que o relacionamento é opcional representado por 0;
				- Obrigatória: indica que o relacionamento é obrigatório representado por 1.

	- CARDINALIDADE MÁXIMA: - 1 (UM) E muitos, representado pela letra 'n'.
		[A REPRESENTAÇÃO DAS CARDINALIDADES 1:1, 1:N, N:N são considerados VALORES MÁXIMOS.]
OBS IMPORTANTE:	
		QUANDO A CARDINALIDADE MÍNIMA É OMITIDA, SUBENTENDE-SE QUE ELA PODE SER OPCIONAL(ZERO)

FCC-2018-SEFAZ:
	Uma adequada modelagem de dados é necessária antes da construção dos BD para que estes sejam suficientemente consistentes enquanto fontes de consulta
pela fiscalização.
Um Modelo de dados-exemplo para atender o controle de arrecadação tributária contém:

	- CONTRIBUINTE (Pessoa física ou Jurídica):
		Dados dos contribuintes como:
			- CPF ou CNPJ (chave)
			- Endereço-Contribuinte
	- ARRECADAÇÃO:
		Dados de arrecadação de tributo exigível como:
			- tipo_tributo
			- objeto_tributo
			- num_ocorrencia_pagamento
			- mes_anos_competencia
			- valor_tributo
			- data_vencimento
			- data_pagamento

		REGRAS DE NEGÓCIO:
			- tipo_tributo: é o que identifica um tributo (IPVA,ICMS)
			- objeto_tributo : é um número de identificação sobre o qual incide o tributo(matrícula,renavam,inscrição.estadual)
			- num_ocorrencia_pagamento : é um número sequencial dentro do ano, usado no caso de cotas ou parcelamento do mesmo tributo.
			- Contribuinte e Arrecadação relacionam-se em um-para-muitos cujo relacionamento tem o nome de EXIGIVEL.

Um tipo característico de relacionamento para o qual se deve prestar especial atenção no MER é:
	a) Agregação porque o relacionamento entre Contribuinte e Arrecadação se caracteriza como entidade associativa que irá, por sua vez, relacionar-se
com instâncias correspondentes a outras parcelas de arrecadação.[ERRADO]
[Nas regras de negócio do enunciado da questão nos mostra o atributo 'tipo_tributo' que identifica qual é o tipo de tributação, sem a necessidade de criar
uma entidade associativa. E também tem o atributo 'num_ocorrencia_pagamento' em caso de parcelamento.] 	
[Há também o relacionmento EXIGIVEL para tal fim com relacionamento 1:N]

	b) Autorelacionamento da Entidade Arrecadação, no caso de associação entre parcelas do mesmo tributo.[ERRADO]
[No caso de associação entre parcelas do mesmo tributo temos o atributo 'num_ocorrencia_pagamento' único da entidade. ]
[As entidades Contribuintes e Arrecadação devem obrigatoriamente se relacionarem por serem grau 2 (binário) e cardinalidade 1:N.]
	
	c)Contribuinte como Entidade Fraca de Arrecadação porque esta última é quem determina a existência, ou não, da primeira.[ERRADO]
[A entidade CONTRIBUINTE não pode ser dependente de ARRECADAÇÃO, AMBAS SÃO INDEPENDENTES, uma é física outra é abstrata. No contexto não há como.]
	
	d)Herança Não Exclusiva porque alguma instância da superclasse Contribuinte pode ou não estar associada a uma instância correspondente de Arrecadação.
[ERRADO][Herança Exclusiva porque o contribuinte sempre  estará associado a uma instância da entidade Contribuinte]
	
	e)Generalização/Especialização TOTAL porque toda instância da superclasse Contribuinte precisa estar associada a uma instância correspondente de suas subclasses(física ou jurídica).
[CERTO][ A GENERALIZAÇÃO/ESPECIALIZAÇÃO TOTAL GARANTE UMA HIERARQUIA DE CLASSE CONTRIBUINTE, SENDO QUE HÁ DOIS TIPOS ESPECÍFICOS DE ATRIBUTOS.
		ASSIM CADA CLASSE DEVE MANTER SEU ATRIBUTO ESPECÍFICO E OS DEMAIS, QUE SÃO COMUNS, NA CLASSE SUPERIOR]


FCC-2016-PREFEITURA:
	-Um analista de Sistemas da Prefeitura de Teresina necessita modelar em um DER as consultas dos cidadãos aos processos públicos.
O relaciomento da consulta definido é N:M(muitos-para-muitos). Todavia cada consulta realizada deve ter uma identificação própria e mais o atributo
data da consulta. Este relacionamento assim especificado relaciona-se com outras entidades do modelo, de acordo com o levantamento de requisitos.
	Pela característica assim definida, esse relacionamento de consulta deve ser desenhado como:

a)entidade associativa = 	O relacionamento entre CIDADÃO E PROCESSO tem a cardinalidade N:M, assim deve ser 'quebrado' em dois relacionamentos 1:N
e assim realizar a associação dos registros dos cidadãos com seus processos, acrescentando o atributo DATA para distinguir a associação dos registros.

	UMA ENTIDADE ASSOCIATIVA É A REDEFINIÇÃO DE UM RELACIONAMENTO, QUE PASSA A SER TRATADO COMO SE FOSSE TAMBÉM UM ENTIDADE E ASSIM
					SE RELACIONAR A UMA ENTIDADE OU A OUTRO RELACIONAMENTO.
