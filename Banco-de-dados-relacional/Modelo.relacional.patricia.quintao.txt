MODELAGEM NÍVEL LÓGICO - RELACIONAL - PROFESSORA PATRÍCIA QUINTÃO
	- A finalidade global deste modelo é descrever o dados usando um formato tabular padrão(todos os elementos são localizados em TABELAS BIDIMENSIONAIS)
	- O conceito de tabela é mais forte no modelo relacional.
	- Cada uma dessas tabelas é atribuido um NOME ÚNICO.
	- As tabelas organizam os dados em linhas e colunas, simplificando o acesso e a manipulação dos dados.
	- Uma vez colocados os dados no BD relacional utiliza-se as operações de ÁLGEBRA RELACIONAL.
NA TERMINOLOGIA MODELO RELACIONAL:
	- Cada tabela é chamada de RELAÇÃO
	- Uma linha da uma tabela é chamada de TUPLA.
	- O nome de cada coluna é chamado de ATRIBUTO.
	- O tipo de dado que descreve cada coluna é chamado de DOMÍNIO.
	- O DOMÍNIO é um conjunto de valores atômicos, indivisível.
	- As relações tem um GRAU. Grau ou ARIDADE é o número de atributos(colunas) que a mesma possui.

BANCO DE DADOS--->COLEÇÃO DE RELAÇÕES
RELAÇÃO----> UMA TABELA
ÚNICO REGISTRO----> SERÁ CHAMADO DE LINHA(TUPLA)
ATRIBUTO------> SERÁ CHAMADO DE COLUNA
DOMÍNIO----> TIPOS DE VALORES QUE PODEM APARECER EM UMA COLUNA

NO MODELO FÍSICO UMA RELAÇÃO É UMA TABELA
NO MODELO FÍSICO UM ATRIBUTO É UM CAMPO
NO MODELO FÍSICO UMA TUPLA É UM REGISTRO.

ESAF-2002-SRF:
	Em um BD relacional, os objetos que realmente armazenam os dados são as tabelas.

CARACTERÍSTICAS DAS RELAÇÕES:
	- Cada ATRIBUTO de uma relação tem um nome que é único para aquela relação.
( não podem existir dois atributos com o mesmo nome dentro da relação )
		***No entanto, duas relações distintas podem compartilhar o mesmo nome para algum atributo***

TODOS OS ATRIBUTOS DEVEM TER VALORES ATÔMICOS, ÚNICOS.
Sendo assim, não é possível ter atributos multivalorados ou compostos no modelo relacional.
O MER aceitava esses atributos.
TODAS AS TUPLAS DEVEM SER ÚNICAS
- CADA TUPLA DEVE TER NO MÍNIMO UM VALOR DIFERENTE -

No modelo Relacional a ordem das tuplas(linhas)não é relevante.
No modelo Relacional a ordem dos ATRIBUTOS(colunas) é RELEVANTE.
Para o autor C.J DATE uma tupla é um conjunto, não importando os valores.

ORDENAÇÃO -
	A ordenação de TUPLAS em uma relação pode ser definida como um conjunto que são definidas matematicamente como elementos de um conjunto da relação.
*** A ORDEM DELAS NÃO ALTERA EM NADA A RELAÇÃO ***
Sendo assim,uma TUPLA é uma lista ordenada de valores, que não é RELEVANTE. AGORA A ORDEM DAS COLUNAS É RELEVANTE A MENOS QUE A CORRESPONDÊNCIA ENTRE ELAS
E SEUS VALORES SEJA MANTIDA.

									CHAVES
	As tabelas relacionam-se umas às outras por meio de CHAVES. Uma chave pode ser definida como um conjunto de um ou mais atributos que determinam a
UNICIDADE de cada registro.
OBS: A UNICIDADE DOS REGISTROS, DETERMINADA POR SUA CHAVE, É FUNDAMENTAL PARA A CRIAÇÃO DOS ÍNDICES.

TIPOS DE CHAVES:
	SUPERCHAVE: é um conjunto de um ou mais atributos, tomados coletivamente, que nos permitem identificar de forma exclusiva uma tupla na relação.
			Cada relação tem pelo menos uma superchave padrão.
			A superchave especifica uma restrição de exclusividade de que duas tuplas não podem ter os mesmos valores para todos os seus atributos.
			Sendo assim uma superchave pode ter atributos redundantes ou desnecessários para a identificação única de uma tupla.
	CHAVE PRIMÁRIA(PK-PRIMARY KEY): É um atributo ou uma combinação de atributos(colunas) cujos valores distinguem uma linha das demais, dentro de uma tabela.
É um identificador EXCLUSIVO de todas as informações de cada registro dando-lhe UNICDADE. A chave primária nunca se repetirá.
	CHAVE ALTERNATIVA(CHAVE CANDIDATA): é aquela que possui um identificador único que não foi escolhida para ser chave primária podendo ser a mais próxima
a ser nomeada a ser chave primária.
exemplo: Em uma tabela na qual existem os campos CPF,TITULO_ELEITOR e NUMERO_IDENTIDADE tem-se que todos esses atributos são únicos,eles são chamados de 
CHAVES CANDIDATAS ou CHAVES ALTERNATIVAS. Uma dessas chaves convenientemente será escolhida como chave primária e as demais permanecerão como CHAVES CANDIDATAS.
	CHAVE ESTRANGEIRA(FK-FOREIGN KEY) : É um atributo(coluna) ou combinação de atributos(colunas) cujos valores aparecem na chave primária(candidata) de uma
tabela do banco. é necessária e formada apartir de um relacionamento com a chave primária da outra tabela. Ela permite navegar em um banco de dados relacional.
UMA TABELA PODE TER VÁRIAS CHAVES ESTRANGEIRAS E CADA CHAVE ESTRANGEIRA PODE REFERENCIAR UMA TABELA DIFERENTE.

	O MODELO RELACIONAL TEM POR FINALIDADE REPRESENTAR OS DADOS COMO UMA COLEÇÃO DE RELAÇÕES EM QUE CADA RELAÇÃO É REPRESENTADA POR UMA TABELA, COMPOSTA POR
LINHAS,COLUNAS E CHAVES PRIMÁRIAS, RELACIONADAS POR MEIO DE CHAVES ESTRANGEIRAS.

CHAVE ESTRANGEIRA - É uma chave primária que vai 'morar' em outra tabela, por isso ela é chamada de estrangeira.
			*** Toda chave estrangeira é uma chave primária em alguma tabela e serve para permitir um relacionamento entre tabelas no modelo relacional.
Uma chave estrangeira é aquela que permite uma ligação lógica entre duas tabelas. é um atributo ou conjunto de atributos que se ligam logicamente a chave primária
ou candidata de outra tabela.
Chave estrangeira é um atributo que pode ser CHAVE PRIMÁRIA ou CANDIDATA de outra relação.

RESTRIÇÕES DE INTEGRIDADE:
	O modelo relacional traz consigo uma série de restrições. Que são regras aplicadas pelo BD para garantir que o banco permaneça íntegro,exato e consistente.
RESTRIÇÃO DE INTEGRIDADE DE DOMÍNIO:
	Visa garantir que os valores que cada atributo irá receber estejam dentro do seu domínio.
(ESSA RESTRIÇÃO VAI AVALIAR OS VALORES DE UMA DETERMINADA COLUNA OU ATRIBUTO PARA CHECAR SE ELE PERTENCE AO DOMÍNIO DEFINIDO NA CRIAÇÃO DA RELAÇÃO.)
	INTEGRIDADE DE VAZIO:
			Subtipo da integridade de domínio, verifica se um campo pode ou não receber NULL em uma coluna da tabela.
INTEGRIDADE DE CHAVE(RESTRIÇÃO DE CHAVE):
			Em uma relação deve ter pelo menos uma chave. A existência de pelo menos uma chave é uma condição OBRIGATÓRIA.
RESTRIÇÃO DE ENTIDADE:
	Especifica que nenhum valor de chave primária pode ser nulo. Uma tupla em uma tabela que se refere a uma outra relação deve referenciar uma tupla 
existente naquela relação. 
INTEGRIDADE REFERENCIAL:
	Visa garantir que o valor de um campo que é chave estrangeira em uma tabela exista na chave primária da tabela de origem.
ASSIM O VALOR DA CHAVE ESTRANGEIRA SEMPRE QUE EXISTIR DEVE ESTAR ASSOCIADO A UM VALOR DA CHAVE PRIMÁRIA DA OUTRA RELAÇÃO. PODENDO ASSUMIR VALOR NULL

INTEGRIDADE SEMÂNTICA:
	As restrições de integridade semântica ou regras de negócio são implementadas por meio de TRIGGERS(gatilhos) e STORED PROCEDURES(procedimentos armazenado no banco)

Resumo:
	Restrições de integridade :
		- Regras aplicadas pelo BD para garantir que ele permaneça íntegro,exato e consistente.
( Elas resguardam o BD contra danos acidentais, assegurando que mudanças não resultem na perda da consistência dos dados. )
	Integridade de domínio[restrição de domínio]:
		- Visa garantir que os valores que cada atributo irá receber estejam dentro do seu domínio.
	Integridade de vazio:
		- Subtipo da integridade de domínio,verifica se um campo pode ou não receber NULL.
	Integridade de chave:
		- Impede que uma chave primária se repita.
		- Uma relação deve ter pelo menos uma chave.
		- A existência de pelo menos uma chave é uma condição OBRIGATÓRIA.
		- Tendo-se em vista que uma relação é definida como um conjunto de tuplas e todas elas devem ter um valor distinto.
	Restrição de ENTIDADE:
		- Especifica que nenhum valor de chave primária pode ser nulo.
	Integridade Referencial:
		- Visa garantir que o valor de um campo que é chave estrangeira em uma tabela exista na chave primária na tabela de origem.

	A ESPECIFICAÇÃO DE UM BD RELACIONAL DEVE CONTER NO MÍNIMO A DEFINIÇÃO DO SEGUINTE:
		TABELAS QUE FORMAM O BD
		COLUNAS QUE AS TABELAS POSSUEM
		RESTRIÇÕES DE INTEGRIDADE
		FORMA-SE ASSIM O ESQUEMA

Lembre-se de que o cabeçalho de uma tabela contém os ATRIBUTOS ou CAMPOS.
Cada ATRIBUTO contém um NOME, TIPO e ORDEM NA TABELA. Isso gera o conceito de ESQUEMA.
Além disso, o esquema deve descrever se um certo atributo deve ser chave ou não para realizar o relacionamento com outras tabelas.

***Dessa forma, somente o cabeçalho de uma tabela NÃO CONFIGURA UM ESQUEMA.***

REGRAS DE CODD PARA BD RELACIONAL
	Essas regras tem o objetivo de definir o que é necessário para que um sistema de gerenciamento de BD seja considerado RELACIONAL.
obs:* As doze regras de Codd são na verdade um conjunto de treze regras(são enumeradas de zero a doze.)

REGRA 0: 
Regra fundamental ou regra base:
	- Um SGBD relacional deve gerenciar seus dados usando exclusivamente capacidades relacionais.
REGRA 1:
	- A regra da informação ou princípio da informação:
		Todas as informações de um BD relacional devem ser representadas logicamente como valores de coluna em linhas dentro das tabelas.
REGRA 2:
	- Regra de garantia de acesso:
		Cada um e qualquer valor atômico(datum) possui garantia de ser logicamente acessado pela combinação do nome da tabela, 
do valor da chave primária e do nome da coluna.

REGRA 3:
	- Tratamento sistemático de valores nulos:
		- O SGBD deve permitir que cada campo possa permanecer nulo(vazio). Os nulos devem ser representados e tratados de modo sistemático,
independentemente do tipo de dados.
REGRA 4:
	- Catálogo online dinâmico baseado no Modelo Relacional:
		- Os metadados devem ser armazenados e gerenciados como dados comuns, em tabelas no inteiror do BD. Esses dados devem estar disponíveis aos 
usuários autorizados, utilizando a linguagem relacional padrão do BD.
REGRA 5:
	- Sub linguagem ampla de dados:
		- O BD relacional pode suportar várias linguagens. No entanto deve suportar uma linguagem declarativa bem definida com suporte para definição de
dados, definição de visualização, manipulação de dados(interativa ou por programa), restrições de integridade, autorização e gerenciamento de transações(iniciar, confirmar e desfazer)
REGRA 6:
	- Atualização de visões:
		- todas as visões que são teoricamente atualizáveis devem ser atualizáveis pelo sistema.
REGRA 7:
	- Inserção,atualização e exclusão de alto nível:
		- o BD deve dar suporte a configuração do nível de inserções, atualizações e exclusões. Isto significa que a capacidade de manipular um conjunto
de dados através de um comando, deve se estender as operações de DML (data manipulation language)
REGRA 8:
	- Independência física de dados:
		- Aplicativos e recursos 'ad hoc' não são afetados logicamente quando os métodos de acesso ou as estruturas de armazenamento físico são alterados.
REGRA 9:
	- Independência lógica de dados:
		- Aplicativos e recursos 'ad hoc' não são afetados logicamente quando de alterações de estruturas de tabela que preservem os valores originais da
tabela(alteração da ordem ou inserção de colunas). Alterações nas relações e nas visualizações (views) causam pouco ou nenhum impacto nas aplicações.
REGRA 10:
	- Independência de integridade:
		- Deve ser possível que todas as restrições de integridade relacional sejam definidas na linguagem relacional e armazenadas no catálogo de 
sistema, não no nível de aplicação. As aplicações não devem ser afetadas quando ocorrer mudanças nas restrições de integridade.
REGRA 11:
	- Independência de distribuição:
		- A distribuição de partes do SGBD em várias localidades deve ser transparente para os usuários. Os usuários não são afetados pela localização 
dos dados (BD distribuídos X BD locais)
REGRA 12:
	- Não transposição das regras:
		- Se o sistema dá suporte a acesso de baixo nível aos dados, não deve haver um modo de negligenciar as regras de integridade do BD.

	FASES DA MODELAGEM DE DADOS:
	- atenção aos termos que definem as fases de modelagem de dados. --- Modelo Conceitual -MER
									 --- Modelo Lógico - MODELO RELACIONAL
									 --- Modelo Físico
	- MER (modelo entidade relacionamento ) corresponde a fase conceitual do projeto do BD. -Visão mais próxima do usuário tem dos dados.
	- Modelo Relacional - Segunda fase do projeto - Corresponde ao modelo LÓGICO.
		- Modelagem a nível lógico
		- Conjunto de relações ou tabelas
		- Algebra relacional
		- Tabelas + restrições de integridade + operadores de manipulação
		- Base do modelo relacional é o uso de tabelas bidimensionais (linhas x colunas)
		- relação (tabela) - Possui GRAU ou ARIDADE que é o número de atributos(colunas) que a mesma possui.	
		- Cada relação terá um nome único e que possui um conjunto de atributos com seus respectivos nomes e domínios.
		- Domínio - Nome dado ao tipo de valor que pode aparecer em cada coluna.
		-Características da relação:
			- ORDENAÇÃO:
			- A ordem das linhas não é relevante.
			- A ordem das colunas é relevante ( a não ser que a correspondência entre elas e seus valores seja mantida )
			ATOMICIDADE:
			- Todos os atributos devem ter valores ATÔMICOS, ou seja, ÚNICOS..
			- Não pode haver atributos MULTIVALORADOS.
			IDENTIDADE:
			- Cada atributo precisa ter um nome único naquela relação.
			UNICIDADE:
			- Todas as tuplas devem ser únicas. Não pode haver 2 ou mais linhas com TODOS atributos iguais.
			(alguns atributos podem ser iguais em mais de uma linha, o que não pode ter é 2 linhas com TODOS os atributos iguais, pois seria a mesma tupla)
			- Atributo é chamado de COLUNA no modelo relacional. 
			- Alguns atributos podem ter valores não aplicáveis ou desconhecidos para algumas situações. São os valores NULOS ou NULL
			- Um registro no modelo físico é chamado de linha(tupla). - Coleção de valores de dados relacionados.
 UM VALOR INDIVIDUAL EXISTENTE NA INTERSEÇÃO DE QUALQUER LINHA COM COLUNA SERÁ CHAMADO DE DADO.
			- Atenção ao conceito de DEPENDÊNCIA FUNCIONAL - Ocorre quando o VALOR assumido por um atributo DETERMINA o valor de outro atributo.
exemplo: O CPF determina o nome de uma pessoa, já o nome não determina o CPF. A chave primária DETERMINA TODOS os demais atributos de uma tupla.
			CHAVES:
			- Chave Primária (Primary Key): -  Uma ou mais colunas com valores que são únicos dentro da tabela e por isso podem ser usados para
identificar as linhas dessa tabela. Por restrição de integridade,os valores das chaves primárias não podem haver valores nulos e nem repetidos.
			- Chave Estrangeira é um conjunto de um ou mais atributos de uma relação que fazem referência a chave primária de outra relação.
			[ATÉ MESMO A PRÓPRIA TABELA NOS AUTO-RELACIONAMENTOS][conexões entre tabelas somente serão expressas através de valores das colunas]
			- Chave Alternativa(chave candidata) = Em certas sittuações mais de uma coluna ou combinação de colunas servem para distinguir uma linha
das demais dentro de uma tabela.

VUNESP-MPE-ANALISTA-2016:

Conceitos: 
		Uma chave primária é um ou mais atributos que representam um registro de forma única em uma tabela. Esse registro também pode ser chamado de TUPLA.
		Uma chave estrangeira é um atributo em uma determinada tabela cujos valores são chave primária em outra tabela. Isso que promove o relacionamento.
		Uma chave candidata é um ou mais atributos em uma tabela que tem potencial de ser uma chave primária, mas não foi eleita para isso.
O fato de ter potencial é que o atributo seria capaz de descrever um registro de forma única na tabela, mas não precisou ser usado para esta função.
Exemplo: a Tabela FUNCIONÁRIO (matrícula,nome, cpf,sexo,rg). A chave primária pode ser considerada a 'matrícula', porém, CPF e RG seriam capazes também de ser
um atributo chave primária porque descrevem de forma única um registro(valor sem repetição)

Em 1985, Edgar Codd publicou as regras de definição dos SGBDS RELACIONAIS. Essas regras servem como modelo de referência sobre o que deve ser um BDR.
REGRA ZERO:
	Todas as regras baseiam-se no conceito de que para que um BD seja considerado RELACIONAL, deve utilizar recursos RELACIONAIS exclusivamente para seu
gerenciamento.
REGRA UM: INFORMAÇÃO (PRINCÍPIO DA INFORMAÇÃO):
		- Todas as informações de um BDR devem ser representadas logicamente como valores de coluna em linhas dentro das tabelas.
REGRA DOIS: GARANTIA DE ACESSO:
		- Deve-se garantir que todos os valores de uma tabela possam ser acessados por meio de uma combinação de nome da tabela, valor da chave primária
e nome da coluna.
REGRA TRÊS: TRATAMENTO DE NULOS - 
		- Os nulos devem ser representados e tratados de modo sistemático, independentemente do tipo de dados.
REGRA QUATRO: CATÁLOGO ONLINE DINÂMICO COM BASE NO MODELO RELACIONAL:
		- Os metadados devem ser armazenados e gerenciados como dados comuns, ou seja, em tabelas no interior do BD. Esses dados devem estar disponíveis
aos usuários autorizados, utilizando a linguagem relacional padrão do BD.
REGRA CINCO: SUBLIMGUAGEM AMPLA DE DADOS:
		- O BDR pode suportar várias linguagens. No entanto deve suportar uma linguagem declarativa bem definida com suporte para definição de dados,
definição de visualização, manipulação de dados(interativa ou por programa), restrições de integridade, autorização e gerenciamento de transações(iniciar,confirmar e
desfazer).
REGRA SEIS: ATUALIZAÇÃO DE VISUALIZAÇÃO:
		- Qualquer visualização que teoricamente possa ser atualizada deve ser por meio do sistema.
REGRA SETE: INSERÇÃO,ATUALIZAÇÃO E EXCLUSÃO DE ALTO NÍVEL - 
		- O BD deve dar suporte a configuração do nível de inserções, atualizações e exclusões. Ter a capacidade de manipular um conjunto de dados 
através de um comando, deve-se estender as operações de DML(data manipulation language) como INSERT,UPDATE e DELETE.
REGRA OITO: INDEPENDÊNCIA FÍSICA DE DADOS -
		- Aplicativos e recursos 'ad hoc' não são afetados logicamente quando os métodos de acesso ou as estruturas de armazenamento físico são alterados.

'ad hoc' --> Termo em latim usado no jurídico para comunicar uma condição temporária que remete a uma conclusão rápida, específica e provisória.
		Uma consulta ad hoc em BD é um tipo de consulta SQL criada no momento em que surge uma necessidade, a partir de requisito específico
			e que não é salva no SGBD(diferente dos procedimento armazenados)para que não seja reutilizada posteriormente.
		ESSE TIPO DE CONSULTA NASCE DE SOLICITAÇÕES NÃO-PLANEJADAS, CUJA NECESSIDADE NÃO FORA PREVISTA COM ANTECEDÊNCIA, SURGINDO DE MOMENTO.

NO GERAL AS SOLICITAÇÕES NÃO-PLANEJADAS SÃO CARACTERÍSTICAS DE APLICAÇÕES PARA APOIO À DECISÃO, COMO APLICAÇÕES PARA BI(BUSINESS INTELLIGENCE)
			OU DATA MINING(MINERAÇÃO DE DADOS) E AS CONSULTAS AD-HOC SÃO BEM COMUNS NESSES SISTEMAS.
**** se a consulta ad hoc for criada usando uma string dinâmica(script PHP ou PYTHON) podemos chamá-la simplesmente de CONSULTA DINÂMICA.****
***** As consultas dinâmicas são consideradas consulta ad-hoc porque também não consistem em declarações armazenadas****

REGRA NOVE: INDEPENDÊNCIA LÓGICA DE DADOS:
		- Aplicativos e recursos 'ad hoc' não são afetados logicamente quando de alterações de estruturas de tabela que preservem os valores originais
da tabela(alteração da ordem ou inserção de colunas). Alterações nas relações e nas visualizações(views) causam pouco ou nenhum impacto nas aplicações.
REGRA DEZ: INDEPENDÊNCIA DE INTEGRIDADE :
		- Deve ser possível que todas as restrições de integridade relacional sejam definidas na linguagem relacional e armazenadas no catálogo do sistema
e não no nível da aplicação. As aplicações não devem ser afetadas quando ocorrer mudanças nas restrições de integridade.
REGRA ONZE:INDEPENDÊNCIA DE DISTRIBUIÇÃO:
		- Os usuários finais e aplicativos não conhecem nem são afetados pela localização dos dados(BD distribuidos VS BD locais).
REGRA DOZE: NÃO TRANSPOSIÇÃO DAS REGRAS:
		- Se o sistema dá suporte a acesso de baixo nível aos dados, não deve haver um modo de negligenciar as regras de integridade do BD.

RESTRIÇÕES DE INTEGRIDADE: São regras de consistência de dados que é ganrantida pelo próprio SGBD.
	- Sâo garantia automaticas pelo SGBDR
	- Integridade de domínio:
		- Define os valores que podem ser assumidos pelos campos de uma coluna.
	- Integridade de Vazio:
		- Especifica se os campos de uma coluna podem ou não serem vazios.
	- Integridade de chave:
		-Define que os valores da chave primária e alternativa devem ser únicos.
	- Integridade Referencial:
		- Define que os valores do campos que aparecem numa chave estrangeira devem aparecer na chave primária(candidata) da tabela referenciada.
Esquematizando:
	- Restrições de Integridade: Regras aplicadas pelo BD para garantir que ele permaneça íntegro,exato e consistente.
		- Resguardam o BD contra danos acidentais,assegurando que mudanças não resultem na perda da consistência dos dados.
	- Restrição de Domínio: Visa garantir que os valores que cada atributo irá receber estejam dentro do seu domínio.
	- Integridade de Vazio: Subtipo da integridade de domínio, verifica se um campo pode ou não pode receber NULL.
	- Integridade de chave: Impede que uma chave primária se repita.
				Uma relação deve ter pelo menos uma chave.
				A existência de pelo menos uma chave é condição obrigatória, tendo-se em vista que uma relação é definida
como um conjunto de tuplas e, todas as tuplas devem ter um valor distinto.
	- Restrição de Entidade:
				Especifica que nenhum valor de chave primária pode ser nulo.
	- Integridade Referencial:
		- Visa garantir que o valor de um campo que é chave estrangeira em uma tabela exista na chave primária na tabela de origem.
	- Integridade Semântica:
		- As restrições de integridade semântica ou regras do negócio são implementadas por meio de GATILHOS ou STORED PROCEDURES.
